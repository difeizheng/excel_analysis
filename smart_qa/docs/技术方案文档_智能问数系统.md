## 1. 项目背景与目标

### 1.1 背景

三峡国际经营数据库以 Excel 形式承载公司历年财务、装机、发电量数据。业务人员希望用**自然语言直接提问**（如"2025 年风电发电量是多少"），系统返回准确结果。

### 1.2 目标（三大硬性要求）

| 要求 | 定义 | 验收标准 |
|---|---|---|
| **准确** | 返回数值与源数据一致 | 6 个测试用例数值 100% 正确 |
| **可追溯** | 每个数值能溯源到具体单元格 | 输出含单元格地址 + 计算链 |
| **零幻觉** | 不确定时拒绝，而非编造 | 找不到数据时返回"无法定位"而非猜测值 |

### 1.3 非目标（明确排除）

- ❌ 不做"大模型直接读表聊天"——精度不达标
- ❌ 不做基于向量检索的 Table-RAG——对聚合/计算类问题幻觉率高
- ❌ 不做自由文本生成——答案必须是"数值 + 单位 + 溯源"，非自然语言长篇

## 2. 需求分析（从 6 个测试用例提炼）

### 2.1 测试用例与能力分类

| # | 问题 | 能力类型 | 隐藏的业务规则 |
|---|---|---|---|
| 1 | 2018 年利润总额 | 单点取数 | 实体消歧："公司/三峡国际" vs "集团" → 不同行 |
| 2 | 2022/2024/2025 汇兑净损失 | 非连续多年取数 | 稀疏列定位 |
| 3 | 24 年-26 年 2 月累计分红 | 范围求和 | 月度列 = 当年累计(YTD)，取最新月，不累加 1..n 月 |
| 4 | 近三年利润增长率 | 复合计算 CAGR | "近三年"=查询日前 3 年；利润默认=利润总额；期初取前一年底 |
| 5 | 2025 总装机/可控装机/利润/发电量 | 跨表取数 | 单位不同须保留；答案引单元格 `T5/T6/Q3/L38` |
| 6 | 2025 风电发电量 | 跨行条件聚合 | "风电"=陆上风电+海上风电；答案显式给 `L14+L16+...` |

### 2.2 提炼出的功能需求

- **F1 单点取数**：按 (实体, 指标, 时间) 定位单一单元格
- **F2 多年取数**：连续 / 非连续年份批量取数
- **F3 范围求和**：跨年/跨月累计，遵守 YTD 规则
- **F4 复合计算**：CAGR、同比、环比等
- **F5 跨表取数**：一次提问命中多张表，单位不混淆
- **F6 条件聚合**：按分类（风电/水电/光伏）跨行求和，含子类归并

### 2.3 提炼出的业务规则需求（必须固化，不可交 LLM 推理）

- **R1 实体消歧**：公司/三峡国际 → 三峡国际行；集团/集团公司 → 集团行
- **R2 月度 YTD**：2026 月度列为当年累计值，"1-2 月"取 2 月列，禁止 1 月+2 月
- **R3 近三年语义**：锚定查询日，取前 3 个完整年；CAGR 期初取前一年底值
- **R4 默认口径**：未指明时"利润"= 利润总额；"公司"= 三峡国际
- **R5 分类归并**：风电 = 陆上风电 ∪ 海上风电；装机 = 可控 + 权益
- **R6 单位强约束**：装机=万千瓦、财务=亿元、发电量年度=亿千瓦时、月度=万千瓦时

### 2.4 溯源需求

- 每个返回数值必须附带：**来源 Sheet、行语义键、列语义键、物理单元格地址（如 `发电量!L14`）、原始值、单位**
- 复合计算须展示**计算链**（公式 + 每个操作数的来源单元格）

## 3. 设计原则与防幻觉总纲

### 3.1 三条铁律

1. **LLM 不算数**：大模型只负责"理解意图 + 选址"，所有数值运算在确定性引擎完成。
2. **接地或拒绝（Grounding or Refuse）**：任何概念若无法映射到具体单元格，系统返回"无法定位 + 缺什么"，绝不编造。
3. **单元格地址即发票**：每个数值都有一张可核验的"发票"（单元格地址），用户可点击核验。

### 3.2 五大防幻觉机制

| 机制 | 做法 | 挡住的幻觉 |
|---|---|---|
| M1 LLM 不参与算术 | structured output 只出意图 JSON | 心算错、编造数 |
| M2 接地或拒绝 | 映射失败 → 返回缺口，不猜 | 无中生有 |
| M3 规则外置 | 业务规则在语义层 YAML，LLM 引用不推理 | 月度 YTD / 近三年等规则错 |
| M4 单位强类型 | 每个值带单位，运算前校验单位匹配 | 量级错 |
| M5 双重校验 | 引擎重算 + 单元格内容匹配复核 | 静默错误 |

### 3.3 失败模式（Fail-Safe）

系统在以下情况必须**显式降级**而非输出错误答案：
- 指标无法识别 → "未找到指标「XX」，请确认或补充"
- 时间无法定位 → "未找到「XX 年」数据"
- 单位不匹配 → "单位不兼容，无法计算"
- 校验不通过 → "结果存疑，已标记，请人工核验"

## 5. 各模块详细设计

### 5.1 预处理层（①）

**职责**：把 Excel 解析成统一的**带地址单元格网格**（Grid）。**schema 驱动**——Excel 的"长相"（表头行、标签列、数据列范围、小计行分类）全部外置为 YAML，加载器与具体文件解耦：换文件只改 schema，不改代码。

#### 5.1.1 数据结构（不可变契约，下游硬引用）

```python
@dataclass
class Cell:                    # 预处理产出的最小单元
    value: Any                 # 数值(float) 或 文本(str, 如 "转至参股")
    addr: str                  # 物理地址, 如 "发电量!L14"  ← 溯源凭证
    numeric: bool              # 是否可参与计算
    row_idx: int               # 0-based 行号(地址反推用)
    col_idx: int               # 0-based 列号

@dataclass
class Grid:                    # 预处理总产出, 下游唯一的取数入口
    fin: dict                  # {label: {colkey: Cell}}          财务数据
    cap: dict                  # {label: {colkey: Cell}}          装机
    gen_projects: list         # [{name, 方式, 区域, values:{colkey:Cell}}]  发电量明细
    gen_subtotals: dict        # {emit_key: {colkey: Cell}}        发电量小计(必有 "合计" 键)
```

- 列键 `colkey` = `"YYYY年"`（年度）或 `"YYYY-MM"`（月度 YTD）。
- 单位**不在 Cell 内**：由语义层 `metrics.yaml` 按指标绑定（见 §7.1），避免预处理懂业务。
- `Cell.addr` 是"接地或拒绝"与"单元格地址即发票"两条铁律的物理载体。

#### 5.1.2 Schema 层级（`schemas/*.yaml` ↔ `schema_spec.py`）

YAML 键与 frozen dataclass 字段 1:1 对应：

```
WorkbookSpec          # path / engine(auto) / version
  └─ SheetSpec[]      # name = locator 派发键("财务数据"/"装机"/"发电量")
       └─ TableSpec[] # target 决定写入哪个 Grid 字段
            ├─ ColumnSpec[]            # role: label/data/classifier/skip
            ├─ SubtotalRule[]          # gen_subtotals: first-wins 小计分类
            └─ detail_classifier_cols  # gen_detail: {name/方式/区域 → 列 idx}
```

`TableSpec.target` 三种模式，统一走 `load_table()` 一段 Cell 构造循环：

| target | 行准入判定 | 写入 | 典型用途 |
|---|---|---|---|
| `row_map` | 标签通过 `skip_labels` / `skip_label_regex` | `grid.fin/cap[label]` | 财务、装机（扁平索引） |
| `gen_detail` | `detail_marker_col_idx` 列**非空** | `grid.gen_projects[]` | 发电量年度明细 |
| `gen_subtotals` | marker 列**空** + `subtotal_rules` first-wins | `grid.gen_subtotals[emit_key]` | 发电量区域/合计小计 |

发电量一张物理表产出两个 Grid 字段，靠**共享行范围、互斥 marker**：明细表收 marker 非空行，小计表收 marker 为空行——任一行只被一个 target 收，天然不重复。

#### 5.1.3 加载器哲学（dumb indexer）

`loader.py` 是**无脑搬运工**，刻意不做：

- ❌ 层级行父级继承（子项 NaN 不向上继承父类别）→ 交给语义层 / 规划层
- ❌ 小计/明细去重推断 → 由 §5.7 校验层 C4 兜底
- ❌ 调用 LLM → 查询时路径永不 `import schema_proposer`
- ❌ 应用业务规则（YTD / 分类归并）→ 那些在 `pipeline.py`

理由：预处理只对"逐格忠实搬运"负责；任何业务判断进语义层，才能保证可测、可换文件、可审计。

#### 5.1.4 工程闸门（让 schema 可演进而不破坏契约）

| 闸门 | 位置 | 作用 |
|---|---|---|
| **V1–V10 校验** | `validate.py` | Sheet 存在 / 表头解析率 ≥ 80% / label 文本密度 / 数据列数值密度 / 列 idx 越界 / 小计行全覆盖等；proposer 候选必跑，`load_grid` 默认不跑 |
| **指纹回归** | `tests/test_fingerprint.py` | 逐格 bit-identical 对比 `tests/golden/测试数据_legacy.json`；任何地址 / 值漂移当场暴露 |
| **engine 自动选择** | `_pick_engine` | `.xls → xlrd`，`.xlsx → openpyxl` |
| **重复标签策略** | `duplicate_label_policy` | `warn`（默认，记 WARNING 仍覆盖，保输出一致）/ `error` / `allow` |

接入新 Excel 两条路：① 手写 `schemas/*.yaml` → 跑 `validate`；② `schema_proposer.py` 离线 LLM 提议（**结构防火墙**：只喂 `<number>/<text>/<date>` 骨架，绝不暴露真实数值）。

### 5.2 语义层（②）

**职责**：承载所有领域知识，作为 LLM 与引擎之间的"契约"。

**组成**（YAML 文件，见第 7 节）：
- `metrics.yaml`：指标字典（名称、同义词、定位规则、单位）
- `taxonomy.yaml`：分类树（风电→陆上+海上）
- `synonyms.yaml`：同义词 + 实体消歧表
- `rules.yaml`：业务规则（YTD、近三年、CAGR、默认口径）

### 5.3 意图解析层（③）

**职责**：自然语言 → 结构化意图。**这是 LLM 唯一介入的环节。**

**输出 schema**（JSON，严格校验）：
```json
{
  "metric": "风电发电量",
  "entity": "公司",
  "dimensions": {"time": "2025年", "region": "全部"},
  "operation": "sum",
  "expected_unit": "亿千瓦时"
}
```

**约束**：
- 只输出 schema 内字段，禁止输出任何数值
- schema 校验失败 → 重试 N 次 → 仍失败则拒绝
- MVP 阶段用**规则解析器**实现（关键词匹配 + 规则库），生产阶段替换为 LLM（Claude Sonnet 4.6，structured output）

**MVP 规则解析器设计**（可插拔，接口相同）：
```
IntentParser（抽象接口）
├── RuleBasedParser   # MVP：关键词 + 规则库，离线可跑
└── LLMParser         # 生产：Claude/GPT structured output
```

### 5.4 规划层（④）

**职责**：意图 → 具体单元格坐标 + 运算。确定性逻辑。

**规划步骤**：
1. 指标解析：`metric` → 通过 metrics.yaml 得到定位规则（哪张表、哪些行）
2. 实体消歧：`entity` → 通过 synonyms.yaml 确定行
3. 分类展开：若指标属分类节点（如"风电"）→ 通过 taxonomy.yaml 展开为子类行集合
4. 时间解析：`time` → 通过 rules.yaml 解析（"近三年"→具体年份；"1-2月"→月度列 + YTD 规则）
5. 生成计划：`{cells: [...], op: "sum"|"cagr"|"lookup", unit}`

### 5.5 执行层（⑤）

**职责**：按计划做确定性运算。LLM 碰不到。

**支持运算**（`engine.py`，纯 Python float 运算）：
- `lookup`：单值返回
- `sum`：求和（含分类归并），求和前做单位维度闸门校验（见 §5.7 C3）
- `cagr`：复合增长 `(期末/期初)^(1/n) - 1`

**取数后端可切换**（`backend.py`，运算永远在 engine 不变）：
- `MemoryBackend`：直接遍历内存 Grid（默认）
- `SqliteBackend`：走 `cells` 表 SQL，适合交叉查询/远程访问
- `backend="both"`：双引擎互验 —— 两条独立取数路径 + 同一 engine 运算，
  value 与 addr 集合一致才 `verified=True`（最强防幻觉，见 §8.2 第 5 步）

> 设计要点：engine 全程鸭子类型（用 `cell.value`/`addr`/`numeric`，unit 走 `getattr` 兜底），
> 故 `CellView` 只需暴露这三属性即可复用全部运算（含单位 dim 闸门）—— SQLite 接入查询链零改 engine。

### 5.6 答案合成 + 溯源层（⑥）

**输出格式**（三件套）：
```
答：39.91 亿千瓦时
计算链：风电发电量(2025) = Σ 风电项目
  = 发电量!L14 (巴西帕尔梅拉风电, 11.116)
  + 发电量!L16 (巴基斯坦第一风电, 0.872)
  + ... 共 8 项
  = 39.905 ≈ 39.91
规则依据：风电 = 陆上风电 ∪ 海上风电（taxonomy.yaml §风电）
```

每个 `地址` 在前端可点击 → 高亮原单元格。

### 5.7 校验层（⑦）

**职责**：输出前的独立复核。

**校验项**（`qa.py`，短路求值，任一不过 → `verified=False`）：
- C1 数值重算：用独立代码路径（纯 Python sum / 手算 CAGR）再算一遍
- C2 单元格回查：按 addr 经 backend 重新读值，确认等于 operand.value
- C3 单位一致：`units.py` 的 6 维度系统（currency/capacity/energy/ratio/percent/pure）——
  ① 预防：`engine.sum_cells` 求和前 assert 同 dim，冲突抛 `UnitDimensionError`；
  ② 复核：C3 用 `from_metric()` 比对 `result.unit` 的 dim 与指标声明
- C4 小计查重：operand.label 不得含"合计/小计"标记（防 taxonomy 展开混入小计行）
- C5 双引擎互验（`backend="both"` 时）：memory + SQLite 两路 value + addr 集合一致

任一不通过 → 标记"存疑"或拒绝输出。

## 6. 数据预处理规范（针对本 Excel）

> 以下规范现以 `schemas/三峡国际经营数据库.yaml` 为**单一事实源**（字段 1:1 对应）；本节作为人类可读对照保留。

### 6.1 财务数据 Sheet

| 项 | 规范 |
|---|---|
| 表头行 | 第 3 行（0-indexed 第 2 行） |
| 列布局 | 第 5–19 列 = 2011–2025（年度）；第 20–24 列 = 2026-01..05（**月度 YTD**） |
| 行键 | 第 2 列"项目"（利润总额、汇兑净损失、向集团分红…） |
| 实体区分 | 三峡国际行 vs 集团行（净资产收益率等指标有两组） |
| 单位 | 亿元 |
| 剔除 | 底部汇率备注行（文本，非数据） |
| **YTD 规则** | 月度列已是当年累计，"1-2 月"取 2 月列，禁止累加 |

### 6.2 装机 Sheet

| 项 | 规范 |
|---|---|
| 表头行 | 第 3 行 |
| 列布局 | 第 5–19 列 = 2011–2025 |
| 行层级 | 一/二 → (一)/(二) → 1/2/3；子项 NaN 的父级类别继承**不在预处理做**（dumb indexer，见 §5.1.3），交语义层 / 规划层按需展开 |
| 分区 | 可控装机 / 权益装机 两大区 |
| 持股比例 | 第 5 列，数值或 "-" |
| 一带一路 | 第 3 列 "√" 标记（分类，非数值） |
| 单位 | 万千瓦 |
| 退化标记 | "−" 表示项目退出/转出，按缺失处理 |
| 小计行 | 合计、可控装机、权益装机、水电/风电/光伏项目 等为小计行，**聚合时只取其一，避免重复** |

### 6.3 发电量 Sheet

| 项 | 规范 |
|---|---|
| **双表** | 左表 = 年度（2019–2025，按项目）；右表 = 2026 月度（按电站）。**必须拆成两个逻辑表** |
| 左表表头 | 第 4 行；列 5–11 = 2019–2025 |
| 左表行键 | 项目名称 + 发电方式 + 区域 |
| 区域小计 | (一)巴西 / (二)南亚 / (三)欧洲 / (四)拉美，为小计行 |
| 单位 | 左表 = 亿千瓦时；右表 = **万千瓦时**（注意不同） |
| 发电方式 | 水电 / 陆上风电 / 海上风电 / 光伏 → 用于分类聚合 |
| 特殊值 | "转至参股" = 项目已不并表，按不可计算处理并标注 |

### 6.4 单元格地址规则

物理地址 = `Sheet名!列字母行号`，如 `发电量!L14`。
- 列字母：0-indexed col → Excel 字母（0=A，11=L，19=T）
- 行号：0-indexed row + 1
- 此规则与测试用例标注一致（`T5/T6/Q3/L38/L14` 等）

## 8. 溯源与校验机制

### 8.1 溯源链结构

每次回答附带一个 `provenance` 对象：
```json
{
  "result": 39.905,
  "unit": "亿千瓦时",
  "operation": "sum",
  "operands": [
    {"address": "发电量!L14", "label": "巴西帕尔梅拉风电", "value": 11.116},
    {"address": "发电量!L16", "label": "巴基斯坦第一风电", "value": 0.872},
    ...
  ],
  "rules_applied": ["taxonomy.风电=陆上风电+海上风电"],
  "verified": true
}
```

### 8.2 校验流程

1. **重算（C1）**：用与执行层不同的代码路径（纯 Python 循环 vs engine）再算一次
2. **单元格回查（C2）**：经取数后端按 `address` 重新读值，确认一致
3. **小计查重（C4）**：确认没有同时计入明细行和小计行
4. **单位校验（C3）**：求和前 dim 闸门（预防）+ 结果 dim 与指标声明比对（复核）
5. **双引擎互验（C5，可选）**：`backend="both"` 时，memory 与 SQLite 两路独立取数 + 运算，
   value 与 addr 集合一致才通过 —— 两套独立代码路径是终极防幻觉手段

## 9. 技术选型

| 模块 | 选型 | 理由 |
|---|---|---|
| 预处理 | pandas + xlrd/openpyxl + PyYAML | schema 驱动；.xls→xlrd ／ .xlsx→openpyxl 自动选；Excel 结构外置 YAML，换文件零改码 |
| Schema 发现（离线） | schema_proposer.py + anthropic（可选） | LLM 只看类型骨架不见真实数值；无 key 安全降级；候选经 validate 闸门 |
| 语义层 | YAML | 可读、可版本化、可评审 |
| 执行引擎 | 纯 Python 运算 + 可切换取数后端（`backend.py`） | engine 鸭子类型，memory/SQLite/both 可切；SQLite 带 addr 溯源，双引擎互验 |
| 意图解析 | 规则解析器（默认）/ LLM（`llm_parser.py`，OpenAI 兼容 + strict JSON 白名单 + 规则回退） | 可接智谱/通义/DeepSeek/Ollama；LLM 只出意图不算数 |
| 向量库 | **不用**（仅同义词匹配可选） | Table-RAG 对聚合查询幻觉高 |
| 校验 | 规则复核 + 可选 LLM-as-judge | 兜底 |
| 前端 | 答案卡 + 可点击溯源链 | 体现可追溯 |

## 11. 评测方案

### 11.1 回归测试集

- 种子：现有 6 个用例
- 扩展：每个能力类型扩到 8–15 个变体（不同年份/实体/指标），目标 50–100 个
- 每个用例标注：问题、期望答案、期望单元格、期望规则

### 11.2 指标

| 指标 | 定义 | 目标 |
|---|---|---|
| 数值准确率 | 数值与期望一致的比例 | ≥ 99% |
| 溯源完整率 | 答案含正确单元格地址的比例 | 100% |
| 拒绝正确率 | 该拒绝时正确拒绝的比例 | ≥ 95% |
| 幻觉率 | 编造数值的比例 | 0 |

### 11.3 评测自动化

- 每次改规则/代码 → 全量跑回归集 → 输出报告
- CI 集成，准确率回退则告警

## 12. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| Excel 结构变动（行/列偏移） | 取错数 | 预处理做语义定位而非硬编码坐标；加结构校验 |
| 业务规则遗漏 | 算错（如 YTD） | 规则库集中管理 + 用例驱动补全 |
| LLM 意图解析错 | 取错指标/时间 | schema 约束 + 校验层 + 拒绝机制 |
| 单位混淆 | 量级错 | 单位强类型 + 运算前校验 |
| 小计/明细重复计入 | 求和翻倍 | 小计行标记 + 聚合时去重 |
| 数据本身有误（如测试值） | 误导 | 溯源让用户可见原始值；不掩盖 |
